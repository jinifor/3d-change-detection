# 3D 포인트클라우드 변화탐지 웹 플랫폼 요구사항 분석서 (v3)

> **개정 이력**
> - v3: 1차(MVP) 완료 후 2차 개발 범위를 프로젝트 목록 조회·이름 수정·삭제 중심의 프로젝트 운영 관리로 재정의, 기존 정확도·운영 고도화 항목은 3차 이후로 이월
> - v2: 정합(co-registration) 설계 강화, 시각화 도구 deck.gl 전환, candidate volume/height 2차 분리, M3C2-EP 2차 분리(MVP는 M3C2 단일 스케일), 업로드 중단·재개 2차 분리, 좌표 정밀도 처리 전략 추가, MVP 처리 파이프라인 단계별 구체화

---

## 1. 프로젝트 개요

### 1.1 목적

드론 LiDAR로 취득한 시계열 LAS/LAZ 포인트클라우드 데이터를 업로드하여 변화 후보를 자동 검출하고, 웹 브라우저에서 즉시 시각화할 수 있는 플랫폼을 구축한다.

### 1.2 주요 목표

* 대용량 LAS/LAZ 파일 업로드
* **두 시기 데이터 정합(co-registration) 및 품질 검증**
* 변화 탐지 자동 수행
* 변화 후보 객체 자동 생성
* 웹 기반 3D 시각화 (deck.gl)
* 작업 진행률 실시간 확인
* Docker 기반 개발 및 배포 환경 제공

### 1.3 v3 핵심 설계 결정

| 항목 | 결정 | 비고 |
| --- | --- | --- |
| 정합 | **MVP에 경량 ICP + RMSE 품질 게이트 포함** | 변화탐지 정확도의 최대 리스크 요인 |
| 시각화 | **deck.gl + COPC 스트리밍** (CesiumJS/Potree 미사용) | maplibre-gl-lidar 활용 가능 |
| candidate volume/height | **3차 이후로 이월** | MVP는 컬럼 nullable, 2차는 프로젝트 운영 관리 우선 |
| 변화탐지 알고리즘 | **MVP는 M3C2 단일 스케일**, M3C2-EP는 3차 이후 | EP는 불확실성 모델 전제 |
| 업로드 중단·재개 | **3차 이후로 이월** | MVP는 presigned PUT/멀티파트 |
| 좌표 정밀도 | **로컬 원점 오프셋 + float64 전 구간 일관 처리** | deck.gl은 METER_OFFSETS |
| 프로젝트 운영 관리 | **2차 우선 개발 범위로 승격** | 프로젝트 목록 조회, 이름 수정, 삭제 |

## 2. 시스템 범위

### 포함 범위 (MVP)

**데이터 업로드**
* LAS
* LAZ
* MinIO presigned PUT / 멀티파트 업로드 (재개 없음)

**데이터 검증**
* 좌표계(CRS) 검증 및 T1/T2 일치 확인
* Bounds(bbox) 겹침 검증
* 포인트 수 / 밀도 / Scale / Offset 검증
* **프로젝트 공통 로컬 원점 결정**

**정합 (신규)**
* 안정면(지면) 추출 (PDAL `filters.smrf`)
* NumPy/SciPy 기반 point-to-point ICP 미세정합 (T2 → T1)
* 정합 잔차 RMSE 산출 및 품질 게이트
* 강체 변환 행렬 저장

**데이터 처리**
* 포인트클라우드 정규화
* COPC 변환
* 처리용 공간 타일 생성

**변화 탐지**
* py4dgeo 기반 **M3C2 (단일 스케일)**

**결과 생성**
* 변화량(거리) 계산
* 변화 후보 추출 (임계값)
* 클러스터링 (DBSCAN)
* 후보 객체 생성 (footprint, 통계 속성)

**시각화**
* **deck.gl 기반 3D 시각화 (COPC 직접 스트리밍)**
* T1 / T2 / 변화 포인트 레이어
* 변화 후보 하이라이트 및 바운딩박스
* 후보 객체 속성 조회 (피킹)

### 포함 범위 (2차 우선 개발)

**프로젝트 운영 관리**
* 프로젝트 목록 조회
* 상태별 프로젝트 필터링
* 완료 프로젝트 재선택 및 결과 재열람
* 프로젝트 이름 수정
* 프로젝트 삭제
  * PostGIS 프로젝트/작업/타일/후보 메타데이터 삭제
  * MinIO `/project/{id}/` prefix 하위 원본·COPC·결과 산출물 삭제
  * 워커 로컬 작업 디렉터리 삭제

### 제외 범위 (2차 이후)

* M3C2-EP (오차 전파)
* candidate volume / 정밀 height 계산
* 업로드 중단 후 재개 (tus)
* ICP 고도화 및 정합 품질 리포트 확장
* 비강체 정합, 다중 스케일 M3C2
* 딥러닝 기반 변화 분류
* 사용자 권한 체계(RBAC), 멀티테넌트
* Kubernetes 배포
* 실시간 드론 스트리밍

## 3. 비즈니스 요구사항

### BR-001
사용자는 두 시점의 포인트클라우드를 업로드할 수 있어야 한다.

**입력**
```text
T1.laz
T2.laz
```

### BR-002
사용자는 변화 탐지 파라미터를 지정할 수 있어야 한다.

**파라미터 (MVP)**
```text
Voxel Size          # corepoint 다운샘플링 간격 (m)
M3C2 Scale          # projection scale, 거리 측정 원기둥 반경 (m)
Normal Radius       # 법선 추정 반경 (m)
Distance Threshold  # 변화로 판정할 최소 거리 (m)
Cluster Size        # DBSCAN eps / min_samples 기준
Registration RMSE Threshold  # 정합 품질 게이트 임계값 (m)
```

### BR-003
시스템은 변화 탐지를 자동 수행해야 한다.

**처리 내용 (MVP)**
* 검증 (CRS / bounds / 포인트 수 / scale·offset)
* **정합 검증 및 미세정합 (ICP + RMSE)**
* COPC 변환
* 처리 타일 생성
* py4dgeo M3C2 분석 (단일 스케일)
* 후보 객체 생성 (DBSCAN)

### BR-004
사용자는 작업 진행 상태를 확인할 수 있어야 한다.

**상태 (v3)**
```text
UPLOADED
VALIDATING
REGISTERING          # 신규
CONVERTING_COPC
GENERATING_TILES
DETECTING_CHANGE
CLUSTERING
PUBLISHING
COMPLETED
FAILED
```

### BR-005
사용자는 변화 후보를 3D 화면에서 확인할 수 있어야 한다.

### BR-006 (신규)
시스템은 정합 품질(RMSE)을 사용자에게 노출하고, 임계값 초과 시 경고 후 진행 여부를 선택하게 해야 한다.

### BR-007 (2차 신규)
사용자는 기존 프로젝트 목록을 조회하고, 완료된 프로젝트를 다시 선택하여 결과를 재열람할 수 있어야 한다.

### BR-008 (2차 신규)
사용자는 프로젝트 이름을 수정할 수 있어야 한다.

### BR-009 (2차 신규)
사용자는 불필요한 프로젝트를 삭제할 수 있어야 하며, 시스템은 DB 메타데이터와 MinIO 산출물, 로컬 작업 디렉터리를 함께 정리해야 한다.

## 8. MVP 처리 파이프라인 (단계별 상세)

> 모든 무거운 연산은 RQ 워커에서 **비동기** 수행. 타일은 병렬 처리 + 체크포인트 단위. 좌표는 **공통 로컬 원점을 뺀 float64**로 처리.

### Step 0 — 업로드 `UPLOADED`
* 입력: `T1.laz`, `T2.laz`
* MinIO presigned PUT 또는 멀티파트 업로드 (재개 없음 — 3차 이후)
* 저장: `/project/{id}/raw/`

### Step 1 — 검증 `VALIDATING`
* 도구: `pdal info --all`, `pdal info --metadata`
* 검사 항목
  * CRS 존재 여부 + T1/T2 좌표계 일치
  * bbox 겹침 비율 (임계 미만이면 FAILED)
  * 포인트 수 / 밀도 비교 (밀도 편차 경고)
  * Scale / Offset sanity
* **프로젝트 공통 로컬 원점 결정**: `origin = floor(min(T1.bbox, T2.bbox) 코너)` → `project.origin_x/y`에 저장
* 실패 시 `FAILED` (사유 기록)

### Step 2 — 정합 `REGISTERING`
* **2a. 안정면 추출**: PDAL `filters.smrf`(또는 `filters.csf`)로 T1·T2 지면 분류 → 변화 없을 것으로 기대되는 안정 표면 추출
* **2b. 다운샘플**: `filters.voxelcenternearestneighbor`(예: 0.5~1.0 m)로 안정면 경량화
* **2c. ICP**: NumPy/SciPy 기반 point-to-point ICP, T2 안정면 → T1 안정면 정렬 → 4×4 강체 변환 추정
* **2d. 변환 적용**: 전체 T2에 변환 적용 → `t2_aligned`
* **2e. 품질 게이트**: ICP 잔차 RMSE 계산
  * `RMSE ≤ threshold` → 진행
  * `RMSE > threshold` → SSE로 경고 이벤트, 사용자에게 진행/중단 선택 요청
* 저장: `project.registration_rmse`, `job.registration_matrix`

### Step 3 — COPC 변환 `CONVERTING_COPC`
* 도구: `pdal translate ... --writers.copc`
* 대상: `t1.laz → t1.copc.laz`, `t2_aligned → t2_aligned.copc.laz`
* Scale/Offset 보존, 투영 CRS 유지
* 저장: `/project/{id}/copc/`

### Step 4 — 처리 타일 생성 `GENERATING_TILES`
* 겹침 bbox에 대해 공간 그리드 생성 (타일 크기 설정값, 예: 100 m)
* 각 타일 → PostGIS `tile` 레코드 생성 (`status=PENDING`, `bbox`)
* 타일 = 병렬 처리 단위 + 체크포인트 (향후 재개 기반)

### Step 5 — 변화탐지 M3C2 `DETECTING_CHANGE`
* 타일별 병렬 RQ 작업
* 처리 (타일 단위)
  1. COPC range read로 타일 bbox(+버퍼) 영역의 T1, T2 포인트 로드
  2. **로컬 원점 차감 → float64 로컬 좌표 변환**
  3. corepoints: T1을 `Voxel Size`로 다운샘플
  4. 법선 추정: `Normal Radius`
  5. py4dgeo **M3C2 단일 스케일** 실행 (`M3C2 Scale` = projection scale)
  6. 출력: corepoint별 거리값
  7. `|distance| > Distance Threshold` → 변화 포인트
  8. 타일별 변화 포인트 저장(중간 산출), `tile.status=DONE`
* 실패 타일은 `FAILED` 표시 후 재시도

### Step 6 — 후보 추출/클러스터링 `CLUSTERING`
* 전 타일 변화 포인트 통합
* scikit-learn **DBSCAN** (`Cluster Size` → `eps` / `min_samples`)
* 클러스터별 후보 객체 생성
  * `geometry`: 지면 투영 footprint 폴리곤 (convex/concave hull)
  * `centroid`, `bbox`, `area`, `point_count`
  * `distance_mean`, `distance_max`
  * `volume`, `height`: **NULL (3차 이후 산출)**
* **로컬 좌표 → 투영 CRS 복원**(원점 가산) 후 PostGIS `candidate` 저장 (프로젝트 SRID)

### Step 7 — 결과 발행 `PUBLISHING`
* `change.copc.laz`: 변화 포인트 (거리값으로 컬러 매핑)
* `change.geojson`: 후보 footprint + 속성
* 저장: `/project/{id}/results/`
* 뷰어용 서빙: `t1.copc.laz`, `t2_aligned.copc.laz`, `change.copc.laz` (MinIO range request) + 후보는 API/GeoJSON

### Step 8 — 완료 `COMPLETED`
* SSE `job_complete`

### 시각화 (deck.gl)
* 레이어 구성
  * T1 Point Cloud (COPC)
  * T2 Point Cloud (aligned, COPC)
  * Change Points (COPC, 거리 컬러)
  * Change Candidates (GeoJSON 폴리곤)
  * Bounding Boxes
* **좌표 처리**: `coordinateOrigin = project.origin`, `COORDINATE_SYSTEM.METER_OFFSETS` → GPU는 작은 상대좌표만 처리(정밀도 지터 방지)
* 피킹: 후보 클릭 → 속성 패널 (API/PostGIS 조회)

---

## 9. 좌표 정밀도 처리 전략 (신규)

큰 좌표(UTM 등)의 float32 정밀도 손실을 방지하기 위해 **"단일 로컬 원점 + float64"** 원칙을 전 구간에 적용한다.

| 구간 | 처리 |
| --- | --- |
| 검증 | 프로젝트 공통 로컬 원점 결정·저장 (source of truth) |
| 처리 (py4dgeo/NumPy) | 연산 전 로컬 원점 차감 → **float64**, 전체 좌표를 float32로 캐스팅 금지 |
| PDAL | Scale/Offset 보존 |
| COPC | LAS 1.4 Scale/Offset 그대로 유지 |
| 시각화 (deck.gl) | `coordinateOrigin` + `METER_OFFSETS`로 로컬 상대좌표 공급 (loaders.gl Float32 정밀도 이슈 회피) |
| PostGIS | double precision, **투영 CRS(원본 SRID) 저장**, 4326 변환은 지도 표시 필요 시에만 |

> M3C2 거리는 차분값이므로, 두 클라우드를 **동일 datum · 동일 로컬 원점 · float64**로 처리하면 정밀도가 보존된다.

## 10. REST API 요구사항

| Method | Endpoint | 목적 |
| --- | --- | --- |
| `POST` | `/api/projects` | 프로젝트 생성 및 T1/T2 업로드 URL 발급 |
| `GET` | `/api/projects` | 프로젝트 목록 조회 |
| `GET` | `/api/projects?status=COMPLETED` | 상태별 프로젝트 목록 필터링 |
| `GET` | `/api/projects/{project_id}` | 단일 프로젝트 메타데이터 조회 |
| `PATCH` | `/api/projects/{project_id}` | 프로젝트 이름 수정 |
| `DELETE` | `/api/projects/{project_id}` | 프로젝트 삭제 |
| `POST` | `/api/projects/{project_id}/jobs` | 변화탐지 작업 시작 |
| `GET` | `/api/projects/{project_id}/assets` | 완료 프로젝트 결과 산출물 URL 조회 |
| `GET` | `/api/projects/{project_id}/candidates` | 변화 후보 목록 조회 |
| `POST` | `/api/jobs/{job_id}/registration-decision` | 정합 품질 게이트 진행/중단 결정 |

### 프로젝트 삭제 정책
* DB: `project` 삭제 시 관련 `job`, `tile`, `candidate` 메타데이터를 함께 삭제한다.
* MinIO: `/project/{id}/` prefix 하위 원본, COPC, 결과 산출물을 삭제한다.
* 워커 로컬 스토리지: `/data/projects/{id}` 작업 디렉터리를 삭제한다.
* 진행 중인 작업이 연결된 프로젝트는 삭제하지 않고 `409 Conflict`를 반환한다.
* 삭제된 프로젝트는 목록, 결과 조회, 후보 조회 대상에서 제외한다.

## 13. 단계별 범위

### 1차 (MVP) — 완료
* LAS/LAZ 업로드 (presigned PUT/멀티파트, 재개 없음)
* 검증 + **정합(ICP + RMSE 게이트)**
* COPC 변환
* py4dgeo **M3C2 단일 스케일** 변화 탐지
* DBSCAN 후보 생성 (volume/height 제외)
* SSE 진행률 + 정합 품질 노출
* **deck.gl 3D 시각화 (COPC 스트리밍)**
* Docker Compose 배포
* 단일 테넌트 + API 키

### 2차 (V2) — 프로젝트 운영 관리
* 프로젝트 목록 조회
* 상태별 프로젝트 필터링
* 완료 프로젝트 선택 및 결과 재열람
* 프로젝트 이름 수정
* 프로젝트 삭제
* 진행 중 작업이 있는 프로젝트 삭제 방지
* 삭제 시 PostGIS 메타데이터, MinIO 객체, 로컬 작업 디렉터리 정리

### 2차 제외
* M3C2-EP
* candidate volume / 정밀 height
* 업로드 중단 후 재개 (tus)
* 딥러닝 기반 변화 분류
* RBAC / 멀티테넌트
* Kubernetes 배포
* 실시간 드론 스트리밍

---

## 14. 향후 확장 계획

### 3차 (V3) — 정확도 · 운영 · 실용성
* ICP 고도화 + 정합 품질 리포트
* **M3C2-EP (오차 전파)**, 다중 스케일
* **candidate volume(DSM 차분) / 정밀 height 산출**
* **업로드 중단 후 재개 (tus / tusd / Uppy)**
* 후보 검토 UI (accept/reject) + 식생·밀도차 오탐 필터
* 룰베이스 객체 분류 (건물/지면/식생)
* 인증·RBAC·멀티테넌트·감사로그
* 대용량(>1GB) out-of-core 처리, 상한 상향
* 보고서 자동 생성, T1/T2/diff 비교 뷰

### 4차 (V4) — 지능화 · 실시간 · 디지털트윈
* 딥러닝 의미론적 변화 분류 (spconv 2.x 기반 Siamese 등, GPU 워커)
* 다중시점(>2 epoch) 시계열 분석
* 실시간/준실시간 드론 수집 연계
* 디지털트윈 · BIM 연계
* Kubernetes 전환, 워커 오토스케일링
